# Использование под высокой нагрузкой

Подобного рода продукт подразумевает использование с довольно высокой нагрузкой
на отдельные части системы, а именно:

- `/decisions/get-flags`
- `/events/receive`

И, как следствие, на:

- хранилище событий
- хранилище флагов

В данный момент реализованы следующие существенные оптимизации:

- набор postgres-индексов (на часто используемые поля для фильтрации)
- хранение событий и выданных решений в ClickHouse (с партиционированием по времени)
- in-memory кэширование на самых горячих путях: выдаче флагов и приёмке событий
- заблаговременная компиляция DSL таргетирования в байткод Python Virtual Machine
  для обеспечения максимальной быстроты проверки правил таргетирования
  (скомпилированные код-объекты хранятся в памяти).
- заблаговременная компиляция JSON схем событий средствами `jsonschema_rs`
  дя обеспечения максимальной быстроты проверки схем событий
  (скомпилированные объекты хранятся в памяти).
- заблаговременная генерация частей ClickHouse запросов из DSL метрик для
  ускорения процесса их получения (в т.ч. и для более быстрого реагирования
  guardrails).

Для того чтобы необходимые, но тяжёлые в работе части не замедляли остальную систему,
они вынесены в отдельные воркеры, которые могут запускаться как в отдельном процессе
(сейчас), так и в отдельных контейнерах в будущем.

Кроме того, отдельные виды сообщений между модулями реализованы через интерфейсы
уведомлений (notifiers), которые сейчас реализованы как функции, непосредственно
вызывающие другие модули. Однако такая организация позволяет очень просто встроить
очередь сообщений в проект.

Для масштабирования проекта в текущем виде необходимо:

- написать реализацию notifiers, публикующую события на шину
- написать обработчики этих событий, вызывать из них то, что ранее вызывалось InstantNotifier
- при масштабировании воркеров необходимо предусмотреть механизмы, защищающие от
  ситуаций, при которых несколько воркеров начинают работу над одними и теми же
  объектами.
