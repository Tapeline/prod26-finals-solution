# Guardrails

## Сущности

### Правило Guardrail

{{ decorated_definition("guard", "Правило") }}

- <guard:ID Правила>
- <stat:ID Метрики> для контроля
- порог срабатывания (>= чего должна стать метрика, чтобы сработало)
- окно наблюдения (время, например, 10 минут)
- действие (Пауза или Откат к контролю)
- архивирован? (да/нет)

### Запись аудита

{{ decorated_definition("guard", "Запись аудита") }}

- <guard:ID Записи> аудита
- <guard:ID Правила>, сработавшего в данном случае
- время срабатывания
- <exp:ID> Эксперимента, для которого сработало правило
- <stat:ID Метрики>, по которой сработало правило
- значение метрики на момент срабатывания
- предпринятое действие (На паузе / Откат к контрольному значению)

## Сценарии

### Создание правила

**Кто**: <access:Экспериментатор>, <access:Администратор>

**Шаги**:

1. Генерация <guard:ID Правила>.
2. Создание правила на основе входных данных.

### Просмотр списка правил

**Кто**: любой пользователь

### Изменение правила

**Кто**: <access:Экспериментатор>, <access:Администратор>

**Ограничения**:

- правило не архивировано

**Шаги**:

1. Изменение метрики, порога, окна и действия на основе входных данных.

### Архивация правила

**Кто**: <access:Экспериментатор>, <access:Администратор>

**Шаги**:

1. Пометить правило как архивированное

### Автоматическая проверка безопасности

**Кто**: Система (фоновый процесс) ??? Раз в сколько ???

**Шаги**:

Для каждого не архивированного правила:

1. Подсчитать метрику
2. Если значение > порога:
    1. Выполнить действие (Статус Эксперимента = На паузе или Остановлен по соображениям безопасности)
    2. Добавить в журнал аудита соответствующую запись
