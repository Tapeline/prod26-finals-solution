# Сбор событий

## Сущности

### Тип события

{{ decorated_definition("evt", "Тип события") }}

- <evt:ID Типа>
- название
- <evt:Схема события>
- <evt:Политика атрибуции>
- архивирован? (да/нет)

### Событие

{{ decorated_definition("evt", "Событие") }}

- <evt:ID> События
- <rt:ID Решения>
- <evt:Тип события>
- временная метка события
- временная метка сохранения события
- дополнительные параметры в соответствии со схемой
- <evt:Статус> обработки

## Объекты-значения

### Схема события

JSON значение в соответствии с [ADR009](../adr/009-event-schema.md).

## Сценарии

### Создание типа события

**Кто**: <access:Экспериментатор>, <access:Администратор>

**Ограничения**:

- <evt:ID Типа> события уникален

**Шаги**:

1. Добавляет новый тип в каталог с данной информацией.

### Просмотр каталога событий

**Кто**: любой пользователь

### Изменение типа события

**Кто**: <access:Экспериментатор>, <access:Администратор>

**Ограничения**:

- нельзя менять <evt:ID Типа> события
- нельзя менять <evt:Схема события>
- нельзя менять <evt:Политика атрибуции>
- тип не архивирован

!!! note
    Нельзя менять <evt:ID Типа>, <evt:Схема события>, <evt:Политика атрибуции>,
    так как изменение "на лету" может привести в рассогласованное состояние
    хранилище событий. Вместо изменения рекомендуется создать новый тип события,
    а этот архивировать.

**Шаги**:

1. Изменить данный тип события

### Архивировать тип события

**Кто**: <access:Экспериментатор>, <access:Администратор>

**Шаги**:

1. Пометить как архивированный

!!! note
    Архивация типа события не имеет никаких последствий для его обработки,
    а скорее служит маркером разработчику больше не использовать этот
    тип события.

### Отправка событий

**Кто**: Субъект

**Ограничения**:

- <evt:Тип события> существует
- payload соответствует схеме типа события

**Шаги**:

1. Для каждого события:

    1. Запросить информацию о <evt:Тип события|Типе события>.
    2. Если такого <evt:Тип события> не существует, пометить как невалидное, конец.
    3. Если payload не соответствует схеме типа события, пометить как невалидное, конец.
    4. Если такой <evt:ID> уже зарегистрирован, пометить как дубликат, конец.
    5. Если событие требует атрибуции, пометить как ожидающее атрибуцию
    6. Иначе пометить как принятое
    7. Сохранить с указанием времени сохранения

2. Вернуть отчёт по всему батчу событий
