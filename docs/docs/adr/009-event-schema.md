# JSON Schema в качестве формата схемы событий

## Контекст и проблема

В платформе A/B тестирования существует настраиваемый «Каталог событий». 
Каждый тип события должен иметь описание ожидаемых данных в поле дополнительных параметров (JSON). 
Необходимо выбрать формат описания этой схемы, который обеспечит:

- высокую скорость валидации
- гибкость
- машиночитаемость для аналитики

## Рассмотренные варианты

- Собственный упрощённый формат (имя поля -> тип данных)
- Бинарные форматы (Protobuf / Avro)
- JSON Schema

## Решение

Выбрана JSON Schema с использованием механизма предварительной компиляции схем 
в памяти API-сервиса. 
Это обеспечивает высокую гибкость описания данных без потери производительности при 
обработке входящего потока.

### Последствия решения

- Хорошо, потому что использование стандарта позволяет аналитикам 
  легко настраивать схемы без изучения специфичных форматов.
- Хорошо, потому что скомпилированный валидатор будет достаточно быстр
- Хорошо, потому что типы данных из JSON Schema (`number`, `string`, `integer`) 
  однозначно транслируются в функции ClickHouse для дальнейшей аналитики 
  (`JSONExtractFloat`, `JSONExtractString`, `JSONExtractInt`).
- Плохо, потому что требует дополнительного шага прогрева кэша
  (компиляции) при старте приложения или обновлении каталога.
- Плохо, потому что при горизонтальном масштабировании потребуются дополнительные
  действия для обеспечения согласованности кэша скомпилированных валидаторов.

### Реализация

Схема хранится в PostgreSQL в поле типа JSONB. 

При запуске система загружает из базы данных схемы и компилирует их. 
При добавлении нового типа он тут же компилируется и сохраняется в кэш.

Схема используется не только для проверки, но и как метаданные для 
генератора аналитических отчётов, например:

1.  Сервис читает `properties` из JSON Schema.
2.  Для поля `amount` с типом `number` формируется SQL-выражение: 
    `JSONExtractFloat(additional_params, 'amount')`.

## "За" и "против"

### Собственный формат (Key-Value)

- Хорошо, потому что очень прост в реализации парсинга.
- Плохо, потому что не позволяет описывать сложные правила 
  (например, "промокод должен соответствовать регулярному выражению").
- Плохо, потому что требует написания собственного валидатора с нуля.

### Двоичные форматы (Protobuf / Avro)

- Хорошо, потому что обеспечивают минимальный размер сообщения и максимальную скорость.
- Плохо, потому что усложняют отладку (данные не читаемы человеком в сыром виде).

### JSON Schema (Стандарт)

- Хорошо, потому что является общепринятым стандартом с богатой документацией.
- Хорошо, потому что позволяет валидировать вложенные структуры данных.
- Плохо, если использовать интерпретируемые библиотеки валидации 
  (медленно), однако это решается компиляцией в рантайме.

## Использованные термины

**adr009:JSON Schema**
:   Стандарт описания структуры и ограничений JSON-документов. 
    Позволяет задавать типы данных, обязательные поля и форматы значений.

**adr009:JSONExtract**
:   Семейство функций в ClickHouse для эффективного извлечения типизированных данных из JSON-строк.
