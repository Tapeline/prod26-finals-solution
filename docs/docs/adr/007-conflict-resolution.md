# Разрешение конфликтов в экспериментах на основе политик

## Контекст и проблема

При переходе от простой системы фича-флагов к платформе 
экспериментирования возникает проблема пересечения аудиторий. 
Один пользователь может одновременно подходить под условия таргетинга 
нескольких активных экспериментов в одной функциональной области.

Одновременное участие в нескольких изменяющих один и тот же функционал тестах приводит к 
искажению данных из-за взаимного влияния, что делает результаты неинтерпретируемыми. 
Необходимо определить механизм, который будет детерминировано решать, 
какой эксперимент показать пользователю, обеспечивая чистоту данных.

## Рассмотренные варианты

- Полная ортогональность
    
    Деление трафика на изолированные слои, позволяющее пользователю участвовать в экспериментах независимо.
 
- Политика на уровне Эксперимента

    Каждый эксперимент сам определяет, как он взаимодействует с другими при конфликте

- Политика на уровне Домена

    Домен (как разделяемый ресурс) диктует правила для всех экспериментов внутри себя.

## Решение

Выбрана опция Политика на уровне Эксперимента с поддержкой двух режимов: 

- один или никто — строгое исключение
- наивысший приоритет

Это решение принято исходя из требований ТЗ и простоты реализации.

### Последствия решения

- Хорошо, потому что гарантирует чистоту данных, исключает риск 
  смешивания эффектов от разных тестов.
- Хорошо, потому что поведение системы становится детерминированным и 
  предсказуемым для продакт-менеджера.
- Плохо, потому что в режиме "один или никто" уменьшается полезная выборка 
  (мы жертвуем трафиком ради достоверности).
- Плохо, потому что отсутствует механизм сложного пересечения по слоям, 
  который может потребоваться в будущем для высоконагруженных доменов.

### Реализация

1. Выбор политики на основе набора конфликтующих экспериментов:
    выбирается всегда самая ограничивающая. Т.е:

    - `n` x Наивысший приоритет, `0` x Один или никто — выбирается Наивысший приоритет
    - `0` x Наивысший приоритет, `n` x Один или никто — выбирается Один или никто
    - `n` x Наивысший приоритет, `m` x Один или никто — выбирается Один или никто

2. Если политика — Один или никто, все эксперименты отбрасываются.

3. Если политика — Наивысший приортиет, выбирается эксперимент с наивысшим приоритетом.

4. Если таковых несколько, считается tie-breaker хэш (от домена и ID эксперимента) — 
    побеждает тот, у кого хэш больше.

!!! note
    Приоритет с наименьшим числовым значением выигрывает (0 важнее, чем 1).

## "За" и "против"

### Полная ортогональность
- Хорошо, потому что позволяет использовать 100% трафика для разных тестов одновременно.
- Плохо, потому что требует сложного математического аппарата для доказательства 
  отсутствия влияния экспериментов друг на друга. Слишком сложно для MVP.

### Политика на уровне Домена

- Хорошо, так как правило пользования разделяемым ресурсом устанавливается самим ресурсом
- Хорошо, так как избавляет от необходимости выбора применяемой политики

Несмотря на явное преимущество при прочих равных перед выбранным вариантом, ТЗ
явно описывает принадлежность политики разрешения конфликта к эксперименту.

## Использованные термины

**adr007:Один или никто**
:   Политика-"предохранитель". 
    Если пользователь попадает в выборку более чем одного эксперимента в домене, 
    он исключается из всех экспериментов этого домена.

**adr007:Наивысший приоритет**
:   Политика, при которой пользователю показывается только один эксперимент с 
    наивысшим <exp:Приоритет|Приоритетом>.
