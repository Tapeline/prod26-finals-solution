# Выбранный стек

## Контекст и проблема

Для реализации модульного монолита с чистой архитектурой на Python
нужно выбрать веб-фреймворк, способ сериализации/десериализации ответов/запросов
и способ работы с РСУБД. Кроме того, специфика архитектуры предполагает
удобный и мощный Dependency Injection контейнер.

## Рассмотренные варианты

- "Классический" современный асинхронный стек в виде
  FastAPI + Pydantic + SQLAlchemy ORM + dependency-injector.
- Litestar + msgspec + SQLAlchemy Core + Dishka

## Решение

Выбрана связка:

- Веб-фреймворк Litestar
- Сериализатор msgspec
- SQLAlchemy Core (без ORM) для работы с РСУБД
- DI-контейнер dishka

### Последствия решения

- Хорошо, потому что мы получаем максимальную производительность сериализации
  msgspec работает в разы быстрее Pydantic, 
  а SQLAlchemy Core вместо ORM исключает двойной маппинг
  (таблица -> ORM модель -> доменная модель)
- Хорошо, потому что dishka поддерживает scopes, финализацию и автовайринг
- Плохо, потому что экосистема и комьюнити у Litestar и Dishka меньше
- Плохо, потому что использование SQLAlchemy Core вместо ORM
  требует написания большего количества кода

## "За" и "против"

### FastAPI + Pydantic + SQLAlchemy ORM + dependency-injector

Стандартный де-факто стек для новых Python-проектов.

- Хорошо, потому что популярно и имеет большое комьюнити
- Плохо, потому что Pydantic (даже v2) потребляет много ресурсов при 
  массовой сериализации данных
- Плохо, потому что SQLAlchemy ORM провоцирует протекание абстракций либо
  двойной маппинг (таблица -> ORM модель -> доменная модель)
- Плохо, потому что dependency-injector медленнее развивается и хуже 
  поддерживает современную типизацию по сравнению с Dishka.
