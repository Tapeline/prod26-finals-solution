# Оптимизация получения информации о типе события через In-Memory кэш с инвалидацией

## Контекст и проблема

Как описано в [ADR005](005-events-collection.md), при обработке каждого события сначала
нужно получить информацию о его типе:

- существует ли он
- требует ли атрибуции
- какая схема у события

Однако поход в Postgres непозволительно долгий для этого сценария.

## Рассмотренные варианты

- Cache-aside в Redis
- In-Memory Cache-aside с инвалидацией по TTL
- In-Memory Cache-aside с инвалидацией по PubSub
- In-Memory Cache-aside с инвалидацией по внутреннему вызову

## Решение

Решено использовать In-Memory Cache-aside с инвалидацией по внутреннему вызову.

### Последствия

- Хорошо, потому что обеспечивает максимально быстрый доступ к кэшу из всех возможных
  вариантов
- Хорошо, потому что обеспечивает мгновенную инвалидацию
- Хорошо, потому что почти не требует дополнительной настройки
- Плохо, потому что совершенно не годится для горизонтального масштабирования, придётся
  переходить к In-Memory Cache-aside с инвалидацией по PubSub.

Несмотря на указанные ограничения на масштабирование, в рамках MVP — это самый простой
вариант, который сравнительно легко проапгрейдить до In-Memory Cache-aside с инвалидацией по PubSub.

### Реализация

При запросе информации о типе события, если такое событие не содержится в in-memory
кэше, оно будет загружено из Postgres и сохранено в кэш.

При обновлении информации о типе события, оно будет сразу обновлено в In-Memory 
кэше посредством вызова функции в коде приложения.

В случае апгрейда до PubSub, вызов заменится на выпуск события обновления и 
его прослушивание на всех инстансах.

!!! note
    Архивированные типы событий считаются несуществующими.

## "За" и "против"

### Cache-aside в Redis

- Возможно, лучше покажет себя при горизонтальном масштабировании сервиса
- Плохо, потому что накладные расходы на поход в сеть

### In-Memory Cache-aside с инвалидацией по TTL

- Хорошо, потому что обеспечивает максимально быстрый доступ к кэшу из всех возможных
  вариантов
- Плохо, потому что будет задержка обновления, равная TTL, могут начать отклоняться
  хорошие события

### In-Memory Cache-aside с инвалидацией по PubSub

- Хорошо, потому что обеспечивает максимально быстрый доступ к кэшу из всех возможных
  вариантов
- Хорошо, потому что обеспечивает мгновенную инвалидацию
- Плохо, потому что требует дополнительной настройки publisher и subscriber
