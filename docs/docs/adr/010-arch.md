# Модульный монолит с чистой архитектурой

## Контекст и проблема

Платформа имеет сложную предметную область, включающую управление экспериментами, 
расчет метрик, сегментацию аудитории и сбор аналитики. Необходимо выбрать базовую 
архитектуру приложения, которая:

- Обеспечит высокую скорость разработки на старте
- Позволит легко ориентироваться в кодовой базе по мере ее роста
- Избежит излишних инфраструктурных затрат
- Сохранит возможность в будущем безболезненно масштабировать или разделять 
  систему, если нагрузка на отдельные части (например, приём событий или выдача флагов) 
  существенно возрастет

## Рассмотренные варианты

- Традиционный слоистый монолит (Layered / Clean architecture)
- Микросервисная архитектура
- Строгий модульный монолит (полностью изолированные модули с общением только через шину событий)
- Модульный монолит с регулируемыми зависимостями между пакетами

## Решение

Выбран Модульный монолит с регулируемыми зависимостями между пакетами.

Внутри каждого пакета применяется Чистая архитектура 
(domain, application, infrastructure, presentation). 

Пакеты могут напрямую зависеть друг от друга 
(например, вызывать публичные интерфейсы других пакетов), 
при условии строгого соблюдения Acyclic dependencies principle.

Общий, переиспользуемый код (утилиты, базовые абстракции, коннекторы и т.д.) 
выносится в пакет `shared`.

### Последствия решения

- Хорошо, потому что мы получаем один сервис, 
  что радикально упрощает релизный цикл и эксплуатацию
  (нет сети между компонентами, нет распределенных транзакций).
- Хорошо, потому что благодаря Чистой архитектуре внутри пакетов 
  и границам модулей, код легко тестировать, 
  а в будущем любой сильно нагруженный пакет можно относительно 
  легко выделить в отдельный микросервис.
- Хорошо, потому что такой подход проще относительно 
  строго независимых модулей или микросервисов
- Плохо, потому что пакет `shared` рискует превратиться в свалку.
- Плохо, потому что разрешение прямых зависимостей между пакетами 
  требует постоянного контроля, иначе высок риск случайного создания 
  циклических зависимостей и high coupling.

### Реализация

Код разделяется на модули по их предназначению, каждый из которых делится по правилам
Чистой архитектуры + модуль Shared + модуль точки входа (bootstrap).

## "За" и "против"

### Традиционный слоистый монолит (Layered Architecture)

- Хорошо, потому что это самый простой и привычный подход
- Хорошо, потому что легко писать общие транзакционные запросы к БД
- Плохо, потому что по мере роста платформы код будет трудно поддерживать
- Плохо, потому что размываются границы предметных областей.

### Микросервисная архитектура (Microservices)

- Хорошо, потому что сервисы полностью независимы: 
  их можно писать на разных языках, деплоить в разное время и 
  масштабировать отдельно друг от друга.
- Плохо, потому что огромный инфраструктурный оверхед
- Плохо, потому что сильно усложняется бизнес-логика: 
  появляются проблемы распределенных транзакций, 
  согласованности данных и обработки сетевых сбоев.

### Строгий модульный монолит (с изоляцией модулей)

- Хорошо, потому что достигается loose coupling: 
  модуль ничего не знает о других модулях системы.
- Хорошо, потому что переход на микросервисы в будущем будет простым
  — достаточно заменить In-Memory шину на Kafka/RabbitMQ/NATS.
- Плохо, потому что скорость разработки сильно падает из-за большого 
  количества boilerplate-кода (создание событий, обработчиков, дублирование DTO).
