# Алгоритм принятия решений по флагам

## Контекст и проблема

В рамках реализации платформы A/B-тестирования, необходимо выдавать значения флагов,
основываясь на текущих экспериментах. ТЗ определяет такие ограничения на ввод-вывод:

- Запрос имеет <rt:ID Субъекта>, <rt:Атрибуты субъекта> и набор 
  <exp:Ключ флага|Ключей флагов> (батч-запрос).
- Ответ должен иметь на каждый Ключ флага:

    * <rt:ID Решения>
    * значение флага
    * <exp:ID> <exp:Эксперимент|Эксперимента>, если таковой был применён

ТЗ определяет основу алгоритма:

- Если нет активного эксперимента для этого флага, вернуть значение по умолчанию
- Если есть активный эксперимент на этом флаге и он применим к субъекту (применимость
  определяется по таргетированию и лимитам, определяемым далее), то выбрать вариант
  согласно распределению и вернуть его.

И накладывает дополнительные ограничения:

- Детерминизм

    При одинаковой конфигурации (флаги/эксперименты/статусы/правила) и одинаковом запросе
    должен возвращаться один и тот же результат.

  - Липкость (Stickiness)

    Один и тот же <rt:ID Субъекта> не должен получать разные варианты в рамках одного
    эксперимента, пока параметры эксперимента не изменены.

- Идемпотентность и дедупликация

- Ограничивать количество одновременно применённых к субъекту экспериментов

- Периоды отдыха: после серии экспериментов субъект чаще видит базовое поведение

- Прозрачные правила приоритизации, если на субъект претендуют больше экспериментов,
  чем мы можем позволить себе показать, вследствие вышеуказанных ограничений.

Кроме того, очевидно, что запрос значений флагов должен выполняться быстро.

## Рассмотренные варианты

- Хранение всего состояния в базе данных
- Хранение части состояния в базе данных + In-Memory кэш
- Частично stateless реализация на хэшах и бакетах

## Решение

Выбрана Частично stateless реализация на хэшах и бакетах.

Решение сделано в пользу этого варианта, так как он уже довольно
[широко распространён](#additional) 
в существующих платформах A/B тестирования, а так же даёт несравнимые
преимущества с точки зрения оптимизации, по сравнению с другими вариантами.

### Последствия решения

- Хорошо, потому что очень быстро рассчитать нужный флаг для нужного пользователя
- Плохо, потому что алгоритм и система значительно усложняются и требуют отдельного исследования

### Реализация

В момент выдачи пользователю решения по эксперименту, факт решения заносится
в Redis, после чего для этого эксперимента игнорируется любая другая логика, чтобы
достичь Stickiness (липкости) пользователя к варианту. То есть если пользователь
получил значение флага из эксперимента, то последующие те же запросы должны
возвращать то же самое значение флага (за исключением случаев, когда эксперимент
уже не активен).

При запросе значения, система узнаёт в Redis, сколько экспериментов было показано
с момента последнего кулдауна. Если оно превышает определённое конфигурацией
значение, то выставляется временный флаг с TTL равным периоду кулдауна и субъект
теперь считается в кулдауне, а счётчик экспериментов сбрасывается.

В то же время, в In-memory кэше хранится отображение
<exp:Ключ флага> -> активный <exp:Экперимент>.
Этот кэш обновляется с каждым запуском/остановкой эксперимента и
при запуске системы.

После отсеивания:

- пользователей, попадающих в кулдаун
- флагов, не попадающих в эксперименты (не запущен, нет такого)
- экспериментов, конфликтующих между собой (по критериям разрешения конфликтов)

Нужно распределить пользователя в один из вариантов, согласно указанному распределнию:

1. Генерируется строка, состоящая из ключа флага, ID эксперимента и ID пользователя, например,
`buttonColor:9714-034f-aaffcc31-434f472042da:u13942353`.
2. Затем от этой строки берётся быстрый хэш, например, Murmurhash3.
3. Значение хэша нормализуется как `hash_value / max_hash_value`.
4. Вычисляется необходимый бакет (например, берём 10000 бакетов для точности в 
2 знака после запятой у процентов распределения) по формуле:
`round(total_buckets * hash_norm)`
5. Значения аудиторий вариантов (в т.ч. аудитория, не попадающая в эксперимент) расставляются
на отрезке от 0 до количества бакетов в соответствии с их долями.
5. Номер бакета определяет, в какой вариант попадёт пользователь: допустим, номер бакета 49,
есть 20% A, 30% B и 50% вне эксперимента, значит диапазон бакетов A — 0..20, B — 20..50,
вне эксперимента — 50..100. Бакет 49 попадает в диапазон B, значит выбираем этот вариант.

Алгоритм хеширования и выбора бакета детерминирован, отсеивание флагов, не попадающих в 
эксперименты, и разрешение конфликтов также детерминированы. А значит, детерминированность
всего выбора зависит от отсеивания пользователя по времени отдыха.

Время отдыха имеет также чёткие ограничения: применять алгоритм, если не в кулдауне, 
не применять, если в кулдауне. Назначение кулдауна происходит если время последнего кулдауна
превышает заданное в конфигурации значение. Таким образом образуется цикл с периодом, прямо
зависящим только от конфигурации.

## "За" и "против"

### Хранение всего состояния в базе данных

- Хорошо, потому что довольно просто реализовать "в лоб"
- Очень плохо, потому что очень медленно

### Хранение части состояния в базе данных + In-Memory кэш

- Хорошо, потому что всё ещё сравнительно просто реализуется
- Хорошо, потому что сильно сокращает время ожидания при запросе
  благодаря кэшу 
- Плохо, потому что с ростом количества пользователей будет пропорционально
  расти количество используемой оперативной памяти (для сохранения фактов
  присвоения вариантов флагов в рамках эксперимента)

## Использованные термины

**adr002:Кулдаун**
:   Состояние и числовое значение временного промежутка, в течение которого
    пользователь получает только значение по умолчанию.
