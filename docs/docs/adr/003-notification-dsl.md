# DSL подключений к каналам оповещений и триггеров

## Контекст и проблема

Платформа должна уметь отправлять оповещения во множество внешних систем: 
Telegram, Slack, Email, Discord и др.

Каждая внешняя система требует свои параметры для подключения (ID чата в Telegram, 
адрес Email получателя и т.д.)

Также необходимо уметь настраивать триггеры оповещений и их условия.

## Рассмотренные варианты

- Структурированная конфигурация
  
    Хранение конфигурации в JSON-объектах с уникальной схемой для каждого типа 
    (`{ "type": "telegram", "chat_id": "..." }`, 
    `{"when": "on_guardrail", "guardrail": "..."}`)

- Унифицированный DSL наподобие URL

    Использование строкового формата, аналогичного JDBC.

## Решение

Используем унифицированный DSL.

Этот подход позволяет абстрагировать детали подключения и 
хранить конфигурацию любого канала в виде одной строки.

Кроме того, этот подход хорошо себя показал в схожих ситуациях, например, можно
вспомнить строки подключения JDBC.

В качестве триггеров пока разрешаем только изменения в состоянии эксперимента и
срабатывания guardrail. Обосновывается тем, что оповещения необходимы для быстрого реагирования,
а быстрое реагирование требуется при:

- смене статуса эксперимента, чтобы не простаивала работа. Допустим, эксперимент не
  согласовали и отправили на доработку. Автор должен сразу об этом узнать, чтобы
  сразу приступить к исправлениям.
- когда что-то пошло не так, нужно быстро среагировать и проанализировать причину
  провала "по горячим следам".

Иные же случаи не так критичны, поэтому в рамках MVP реализуем только эти два.

### Последствия решения

- Хорошо, потому что есть единый формат хранения.
- Плохо, потому что нужно парсить строки и, возможно, делать urlencode.

### Реализация

Примеры строк подключения:

- `tg://channel_id`
- `email://admin@t.ru`

Примеры триггеров:

- `experiment_lifecycle`
- `guardrail:p95_too_slow`

## "За" и "против"

### Структурированная конфигурация

- Хорошо, потому что есть строгая валидация типов.
- Плохо, потому что есть сложность добавления новых полей.
- Плохо, потому что есть сложность добавления новых интеграций.
